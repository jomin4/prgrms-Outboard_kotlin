## 질문 로그

### 질문1
**Q.** `wiseSayings.first { it.id == id }`가 이해 안 감. 디버깅해도 잘 안 보임.

**A.** 두 가지 원인. (1) 이름 혼동: `val id`(파싱된 목표 번호)와 `it.id`(현재 검사 중인 원소의 번호)는 이름만 같고 다른 값. `it`은 리스트를 순회하며 원소 하나하나를 가리키는 암묵적 람다 파라미터. (2) `first`, `filter`, `map` 등 코틀린 컬렉션 함수는 `inline`이라 F7(Step Into)로도 별도 함수 프레임으로 안 들어가짐(호출부에 코드가 그대로 복사되어 컴파일됨) — 그래서 디버깅이 잘 안 보였던 것.

**트러블슈팅**
- 증상: `first { }` 람다 내부 동작을 디버거로 추적해도 반복되는 느낌이 안 남
- 원인: `inline` 함수라 Step Into가 일반 함수 호출처럼 동작하지 않음
- 해결: `first { }` 호출부가 아니라 **람다 몸통(`it.id == id`) 그 줄에** 직접 중단점을 찍고, F9(Resume)를 반복해서 누르면 같은 줄에서 `it`이 원소마다 바뀌며 여러 번 멈추는 것을 확인 가능. 자바의 for문 + if + break와 동일한 동작임을 수동 트레이스로 설명.

### 질문2
**Q.** `Rq`의 `init` 블록(`command.split("?")` → `action`/`params` 파싱)이 값이 어떻게 처리되는지 구체적으로.

**A.** `"삭제?id=1"` → `split("?", limit=2)` → `["삭제","id=1"]` → `action="삭제"`. `actionAndQuery.size==2`이므로 `if` true 블록 실행(코틀린 `if`는 값을 반환하는 식이라 자바와 달리 바로 `params`에 대입 가능) → `"id=1".split("&")` → `["id=1"]` → `.associate{}`로 각 원소를 `"="` 기준으로 쪼개 `Pair(key,value)`를 만들고 모아서 `Map` 생성 → `params={"id"->"1"}`. 파라미터 없는 `"종료"`는 `split`이 `["종료"]`(1개)를 반환해 `size!=2`이므로 `params=emptyMap()`. 자바로는 `HashMap`+for문+`split("=")`+`put()`에 해당.
